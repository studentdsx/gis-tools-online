# GIS 在线工具合集需求分析报告

更新时间：2026-06-10

## 1. 背景与目标

本系统当前定位为一个 Web 端 GIS 工具合集，已经具备地图浏览、PostGIS 数据接入、本地/服务器文件加载、图层管理、坐标转换、距离/面积测量、地图截图，以及类 Mapshaper 的数据简化与导出能力。下一阶段目标不应简单复制桌面 GIS 的完整复杂度，而应建设一组“打开浏览器即可完成”的高频工具：数据查看、格式转换、投影转换、轻量编辑、空间分析、地图制图、结果导出。

建议产品定位：

- 面向 GIS 初学者、数据工程人员、规划/自然资源/交通等业务人员，提供低门槛在线处理入口。
- 面向开发者，提供 GeoJSON、Shapefile、PostGIS、Mapbox/XYZ 底图等常用 Web GIS 数据调试能力。
- 面向轻量生产流程，覆盖“导入数据 -> 检查坐标 -> 预览样式 -> 分析处理 -> 导出成果”的闭环。

不建议现阶段追求：

- 完整替代 QGIS/ArcGIS Pro 的大型桌面 GIS 能力。
- 大规模栅格分析、海量矢量拓扑计算、复杂制图出版流程。
- 多人协同、权限、项目管理等平台级能力，除非先把单人工具闭环打磨稳定。

## 2. 当前系统能力盘点

### 2.1 已有核心能力

从前后端源码与 README 盘点，当前系统已经具备以下基础：

- 地图浏览：基于 Mapbox GL JS 展示地图，支持卫星/街道等底图。
- 坐标搜索与定位：输入经纬度后飞行定位，并在地图上放置 marker。
- 点击拾取坐标：点击地图后回填经纬度到顶部输入区。
- 底图管理：内置 Mapbox 底图，支持自定义 Mapbox style 与 XYZ 栅格瓦片。
- PostGIS 接入：支持保存连接、测试连接、列出空间表、读取表数据为 GeoJSON。
- PostGIS 投影处理：后端通过 `ST_Transform(..., 4326)` 与 `ST_AsGeoJSON` 输出 Web 地图可用数据。
- 本地/服务器文件：支持 `.geojson`、`.json`、`.shp`，浏览器本地文件读取支持 `.dbf`、`.prj` 配套读取。
- 坐标转换：前端与后端均已有基于 CRS/PRJ 的 EPSG:4326 转换能力。
- 图层管理：添加、显示/隐藏、移除、拖拽排序。
- 测量工具：距离测量、面积测量。
- 地图截图：导出当前地图画布 PNG。
- Mapshaper 页面：支持上传多种 GIS 数据格式，进行 simplify 预览和导出。

### 2.2 当前短板

- “工具合集”的信息架构还不清晰：目前主要是地图主页面和 Mapshaper 页面，缺少统一工具目录、工具分类、最近使用、输入/输出说明。
- 数据编辑能力不足：只能加载和查看，缺少点线面绘制、要素属性编辑、选择/删除/撤销。
- 数据检查能力不足：缺少坐标范围检测、空几何检测、无效几何检测、字段类型检查、重复 ID 检查。
- 空间分析工具不足：已有测量，但缺少 buffer、clip、intersect、union、difference、dissolve、bbox、center、nearest 等高频工具。
- 导出闭环不足：Mapshaper 能导出，但主地图中的图层缺少统一导出入口、导出格式选择、CRS 选择、字段裁剪。
- 图层样式能力不足：目前以固定颜色渲染为主，缺少按字段分类、渐变、符号大小、透明度、标签。
- 属性表能力不足：缺少字段查看、筛选、排序、定位到要素、选择集导出。
- 大数据体验不足：大 GeoJSON 直接上图会卡顿，缺少抽样预览、进度、取消、Web Worker、矢量瓦片化/FlatGeobuf 流式读取策略。

## 3. 外部工具与能力调研

### 3.1 Mapshaper：格式转换、简化、清洗和批处理

Mapshaper 的命令体系覆盖导入、导出、简化、投影转换、字段处理、过滤、dissolve、clip、erase、merge-layers 等能力，适合作为本系统“数据处理工具箱”的重要参考。当前系统已经接入 mapshaper 后端处理文件，建议继续扩展为可视化参数面板，而不是只停留在 simplify。

参考来源：

- https://mapshaper.org/docs/reference.html
- https://github.com/mbloch/mapshaper/wiki/Command-Reference

适合迁移到本系统的能力：

- 格式转换：GeoJSON、Shapefile、TopoJSON、KML、CSV、FlatGeobuf、GeoPackage。
- 几何简化：保留拓扑、选择算法、预览简化比例。
- 数据清洗：修复/清理异常几何、删除空要素、字段整理。
- 图层级处理：合并图层、裁剪图层、擦除图层、按字段 dissolve。
- 命令预览：高级用户可查看或复制等价 mapshaper command。

### 3.2 Turf.js：浏览器端轻量空间分析

Turf.js 提供大量 JavaScript 空间分析函数，包括 buffer、intersect、union、difference、area、length、bbox、center、nearestPoint、pointsWithinPolygon 等，适合 Web 端小中型 GeoJSON 的即时分析。项目当前已依赖 Turf，并用于测量，是继续扩展前端分析工具的低成本路径。

参考来源：

- https://turfjs.org/docs/
- https://turfjs.org/docs/api/buffer
- https://turfjs.org/docs/api/intersect
- https://turfjs.org/docs/api/area

适合迁移到本系统的能力：

- 单图层分析：面积/长度统计、边界框、中心点、质心、缓冲区。
- 双图层分析：相交、裁剪、差集、合并、点落区统计。
- 快速 QA：检测 bbox 是否越界、估算总面积/总长度、抽样预览。

### 3.3 geojson.io：轻量编辑与分享

geojson.io 的价值在于“浏览器内快速创建、编辑、查看、导入导出 GeoJSON”。它代表了一类非常适合 Web 的 GIS 工具：不追求复杂桌面功能，而是把 GeoJSON 编辑、属性查看、格式导入导出做得很顺。

参考来源：

- https://geojson.io/
- https://github.com/mapbox/geojson.io

适合迁移到本系统的能力：

- 地图上绘制点、线、面。
- 选择要素后编辑属性。
- 左右分栏展示地图与 GeoJSON 文本/属性表。
- 导入粘贴 GeoJSON，实时校验并上图。
- 单图层快速导出。

### 3.4 QGIS Processing：专业 GIS 功能基准

QGIS Processing Toolbox 包含大量成熟 GIS 算法，适合作为功能命名和分析类别的专业参照。Web 端不需要一次性实现全部，但可按“矢量几何、矢量叠加、矢量选择、数据管理、制图导出”来规划工具体系。

参考来源：

- https://docs.qgis.org/latest/en/docs/user_manual/processing_algs/qgis/vectorgeometry.html
- https://docs.qgis.org/latest/en/docs/user_manual/processing_algs/qgis/vectoroverlay.html
- https://docs.qgis.org/latest/en/docs/user_manual/processing_algs/qgis/vectorgeneral.html

适合迁移到本系统的能力：

- 缓冲区、质心、线转点、面转线、几何修复。
- 裁剪、相交、联合、差集。
- 按位置选择、按表达式过滤。
- 重新投影、字段管理、导出格式转换。

### 3.5 Web 端适配原则

适合 Web 端优先实现：

- 输入输出明确、参数少、运行时间短的工具。
- 小中型矢量数据处理。
- 需要即时预览的工具，例如样式、简化、缓冲、裁剪。
- 数据格式转换、坐标转换、字段清洗。
- 浏览器本地可完成的隐私友好工具。

不适合首期 Web 端实现：

- 超大数据全量拓扑分析。
- 长时间批处理任务且无队列系统。
- 复杂投影库和罕见 CRS 全覆盖。
- 专业制图排版、复杂网络分析、三维地质建模。

## 4. 目标用户与典型场景

### 4.1 GIS 初学者/课程用户

需求：

- 快速打开 GeoJSON/Shapefile，确认数据位置是否正确。
- 了解坐标系、属性字段、要素数量、面积长度。
- 做简单缓冲区、裁剪、格式转换。

关键功能：

- 文件拖拽导入。
- 坐标系自动识别与提示。
- 一键修复/转换到 EPSG:4326。
- 简单分析工具向导。

### 4.2 GIS 数据工程/开发人员

需求：

- 调试 PostGIS 表、GeoJSON 接口、Mapbox/XYZ 底图。
- 检查几何是否合法、字段是否完整、坐标是否越界。
- 快速导出测试数据。

关键功能：

- PostGIS 表预览和 SQL 条件过滤。
- 属性表、字段统计、几何质量检查。
- GeoJSON 粘贴校验和格式化。
- API/数据源连接管理。

### 4.3 业务人员

需求：

- 在浏览器里叠加多个图层。
- 测量距离面积。
- 按字段渲染和标注。
- 截图或导出一张可汇报地图。

关键功能：

- 图层样式面板。
- 标签设置。
- 图例、比例尺、指北针。
- 高质量截图导出。

## 5. 产品需求分析

### 5.1 产品结构建议

建议从“地图工作台 + 工具箱 + 数据处理页”三层结构组织：

- 地图工作台：用于加载、查看、叠加、选择、编辑和导出当前图层。
- 工具箱：按类别放置独立小工具，用户选择输入图层和参数后生成输出图层。
- 数据处理页：延续 Mapshaper 页面，承担多格式转换、简化、批处理、命令式高级处理。

工具分类建议：

- 数据导入：文件、PostGIS、URL、粘贴 GeoJSON、底图服务。
- 数据检查：坐标系、几何合法性、字段概览、范围检查、重复值检查。
- 数据转换：格式转换、投影转换、字段重命名/删除、坐标精度压缩。
- 空间分析：缓冲区、裁剪、相交、联合、差集、按位置选择。
- 几何处理：简化、平滑、修复几何、质心、边界框、面转线、线转点。
- 制图表达：样式、标签、图例、截图、专题图。
- 导出分享：GeoJSON、Shapefile、KML、CSV、PNG、项目配置 JSON。

### 5.2 主地图工作台需求

#### R1 图层详情与属性表

说明：点击图层后显示图层详情，包括要素数量、几何类型、字段列表、坐标范围、CRS、来源。提供属性表查看、搜索、排序和定位要素。

价值：补齐“看图但看不到数据”的短板，是后续筛选、导出、编辑的基础。

优先级：P0。

#### R2 单图层导出

说明：图层列表提供导出按钮，支持 GeoJSON 首期导出；后续支持 Shapefile、KML、CSV、FlatGeobuf。导出前可选择是否只导出可见/选中要素、是否转换 CRS、是否压缩坐标精度。

价值：形成完整闭环。

优先级：P0。

#### R3 样式配置

说明：支持图层颜色、线宽、点半径、透明度；支持按字段分类渲染、数值渐变渲染；支持标签字段、字号、颜色、避让策略。

价值：业务用户截图汇报、数据对比必需。

优先级：P1。

#### R4 要素选择与框选

说明：支持点击选择、框选、清空选择、反选；选择集可在属性表中联动高亮并导出。

价值：空间分析和编辑的基础。

优先级：P1。

#### R5 地图绘制与编辑

说明：支持绘制点、线、面；支持编辑节点、删除要素、编辑属性；支持撤销/重做。可基于 Mapbox GL Draw。

价值：补齐在线标注、临时数据制作能力。

优先级：P1。

### 5.3 数据导入与管理需求

#### R6 URL 数据源

说明：支持输入 GeoJSON URL、WMS/WMTS/XYZ、Vector Tile URL、ArcGIS REST FeatureServer 只读加载。

价值：Web GIS 常用调试入口。

优先级：P1。

#### R7 数据源连接管理

说明：统一管理 PostGIS、本地文件夹、自定义底图、URL 数据源。支持命名、测试、编辑、删除、最近使用。

价值：当前已有分散能力，需产品化。

优先级：P1。

#### R8 数据导入向导

说明：导入文件后展示格式、大小、字段、CRS、要素数量、坐标范围；如 CRS 缺失则要求用户指定。

价值：减少“加载成功但位置错误”的困惑。

优先级：P0。

### 5.4 数据检查与清洗需求

#### R9 几何质量检查

说明：检查空几何、无效几何、自相交、重复点、坐标越界、几何类型混杂。输出检查报告，并允许定位到问题要素。

价值：GIS 数据处理高频刚需。

优先级：P1。

#### R10 字段检查与字段管理

说明：字段列表、字段类型推断、空值统计、唯一值统计；支持删除字段、重命名字段、保留字段。

价值：导出与分析前的数据整理基础。

优先级：P1。

#### R11 坐标精度压缩

说明：支持保留 5/6/7 位小数，减少 GeoJSON 体积；显示压缩前后大小和精度影响提示。

价值：Web GIS 数据发布常用优化。

优先级：P2。

### 5.5 空间分析工具需求

#### R12 缓冲区 Buffer

说明：选择输入图层、距离、单位、是否 dissolve，生成新图层。

价值：最常用空间分析之一，Turf 可前端实现。

优先级：P0。

#### R13 裁剪 Clip

说明：使用一个 polygon 图层裁剪另一个图层，输出裁剪结果。

价值：高频数据裁剪场景。

优先级：P1。

#### R14 相交 Intersect

说明：两个图层叠加求交，输出交集图层，可保留字段来源。

价值：空间叠加分析基础。

优先级：P1。

#### R15 差集 Difference / Erase

说明：从输入图层中擦除覆盖图层范围。

价值：规划、用地、缓冲区排除场景常用。

优先级：P2。

#### R16 合并 Union / Merge

说明：合并多个图层或多个要素。区分“简单追加 merge”和“拓扑联合 union”。

价值：数据整理常用，但需明确语义。

优先级：P2。

#### R17 Dissolve 按字段融合

说明：按字段值融合 polygon，支持统计字段聚合。

价值：行政区、用地类别、分组统计常用。

优先级：P2。

#### R18 点落区统计

说明：统计点图层落入面图层的数量，可按字段分组。

价值：业务价值高，适合 Web 中小数据。

优先级：P2。

### 5.6 Mapshaper 页面增强需求

#### R19 多工具参数面板

说明：在 simplify 基础上增加 clean、filter、dissolve、clip、erase、proj、rename-fields、drop-fields、merge-layers 等工具参数。

价值：复用已接入的 mapshaper 后端能力，收益高。

优先级：P1。

#### R20 命令预览与高级模式

说明：低代码表单生成 mapshaper command，高级用户可编辑命令并运行。

价值：兼顾易用和专业。

优先级：P2。

#### R21 处理任务反馈

说明：文件上传、处理、导出显示进度、耗时、输入输出大小、日志和错误详情。

价值：提升稳定感与可解释性。

优先级：P1。

### 5.7 制图与分享需求

#### R22 高级截图导出

说明：截图前可设置标题、图例、比例尺、指北针、分辨率、透明背景、纸张比例。

价值：业务汇报高频。

优先级：P2。

#### R23 项目保存/恢复

说明：将当前底图、图层来源、样式、视图范围、工具参数保存为项目 JSON，支持导入恢复。

价值：减少重复配置。

优先级：P2。

#### R24 分享链接

说明：首期可生成本地项目 JSON；后续若有后端存储，可生成短链接。

价值：协作和演示需要。

优先级：P3。

## 6. 非功能需求

### 6.1 性能

- 小数据：浏览器端直接处理，目标 10 万个点或 2 万个面以内保持可用。
- 中数据：采用 Web Worker，避免 UI 阻塞。
- 大数据：推荐后端处理或转换为 FlatGeobuf/矢量瓦片/抽样预览。
- 所有耗时工具必须支持 loading、取消、错误提示。

### 6.2 安全与隐私

- `.env` 中 token 和数据库密码不得提交。
- PostGIS 连接信息需明确本地存储风险；若进入生产，应改为后端加密存储或会话级连接。
- 文件处理要提示本地处理/上传后端处理的区别。
- URL 数据源需防止 SSRF，后端代理加载外部 URL 时必须做白名单或限制。

### 6.3 易用性

- 每个工具都应包含输入、参数、输出三段式布局。
- 输出一律生成新图层，不覆盖原图层，除非用户明确选择覆盖。
- 工具运行前做参数校验，例如 clip 遮罩必须是 polygon。
- 错误提示要说清楚“哪个图层、哪个字段、哪个要素”出问题。

### 6.4 可扩展性

- 建议抽象 `ToolDefinition`：id、名称、分类、输入约束、参数 schema、执行器、输出类型。
- 前端工具优先使用 Turf；复杂格式/拓扑处理走后端 Mapshaper。
- 图层数据模型应统一记录：id、name、sourceKind、geometryTypes、featureCount、bbox、crs、fields、style、visible。

## 7. 模拟专家评审

说明：以下为基于产品、GIS、前端工程、后端工程和数据安全视角的模拟评审意见，并非真实外部专家评审结论。

### 7.1 GIS 产品专家意见

评审结论：方向正确，但需要避免做成“按钮很多但流程不闭环”的工具堆。首要闭环应是导入、检查、处理、样式、导出。

建议：

- P0 先补属性表、导出、缓冲区、导入向导。
- 空间分析工具要用“输出新图层”的一致心智。
- 工具命名尽量贴近 QGIS/ArcGIS 习惯，同时提供中文解释。

### 7.2 GIS 算法专家意见

评审结论：前端 Turf 适合轻量分析，但复杂叠加和大面数据容易出现性能与拓扑问题。Mapshaper/PostGIS 后端处理应作为补充。

建议：

- Buffer、bbox、center、area、length 可优先前端实现。
- Clip、intersect、dissolve 对复杂 polygon 建议优先接 Mapshaper 或后端。
- 所有分析工具必须记录 CRS，距离/面积计算要提示单位和投影假设。

### 7.3 前端工程专家意见

评审结论：当前功能已具雏形，下一步需要统一数据模型和工具框架，避免每个工具各写一套状态。

建议：

- 建立统一图层 store，而不是多个组件内部各自持有关键状态。
- 大数据处理放进 Web Worker。
- 属性表需要虚拟滚动。
- 地图图层样式配置应数据驱动，避免硬编码颜色和 layer id。

### 7.4 后端工程专家意见

评审结论：后端已有 PostGIS、文件、Mapshaper 能力，适合扩展处理任务接口。但要尽早限制文件大小、任务耗时和外部路径访问。

建议：

- 增加任务型 API：创建任务、查询状态、下载结果。
- Mapshaper 错误要结构化返回。
- 文件目录读取必须限制工作目录或配置白名单。
- PostGIS 查询要分页、字段白名单、SQL 注入防护。

### 7.5 数据安全专家意见

评审结论：作为在线工具合集，用户最关心数据是否上传、存在哪里、是否会泄露。必须在交互中说明处理位置。

建议：

- 每个工具标记“浏览器本地处理”或“上传服务器处理”。
- 连接信息、文件路径、token 不进入报告导出。
- 若支持分享链接，必须做访问权限和过期策略。

## 9. 建议迭代路线

### 第一阶段：补齐闭环，2-3 周

目标：让用户能完成“导入 -> 查看 -> 分析一个简单结果 -> 导出”。

范围：

- 工具箱入口。
- 图层详情。
- 属性表只读。
- 单图层 GeoJSON 导出。
- Buffer 工具。
- 图层统计。
- 导入向导基础版。

### 第二阶段：增强专业可用性，3-5 周

目标：让系统从“能看数据”变成“能处理数据”。

范围：

- 样式配置。
- 标签配置。
- 要素选择与选择集导出。
- 几何质量检查。
- 字段管理。
- Clip/Intersect。
- Mapshaper 多工具面板。

### 第三阶段：形成 GIS 工具合集，5-8 周

目标：让系统拥有清晰工具目录和稳定处理体验。

范围：

- 绘制编辑。
- Dissolve、Difference、Merge。
- 高级截图。
- 项目保存/恢复。
- 大文件处理优化。
- 任务日志和错误报告。

## 10. 验收指标

### 10.1 功能指标

- 支持至少 3 类数据来源：文件、PostGIS、URL。
- 支持至少 5 个独立工具：统计、Buffer、Clip、Intersect、字段管理。
- 支持至少 4 种导出：GeoJSON、Shapefile、KML、CSV。
- 主地图中任一加载图层都能查看属性表并导出。

### 10.2 体验指标

- 5MB GeoJSON 导入到预览时间小于 3 秒。
- 运行工具时有明确 loading、错误、完成状态。
- 工具输出自动加入图层列表，并以可识别名称命名。
- 用户不阅读文档也能完成一次 Buffer 并导出结果。

### 10.3 质量指标

- 空几何、无效输入、CRS 缺失、字段不存在等错误有明确提示。
- 对 PostGIS 和文件路径接口有参数校验。
- 所有新增工具至少有 1 组样例数据验证。

## 11. 结论

本系统已有很好的起点：地图主工作台、数据源接入、图层管理、测量、截图和 Mapshaper 文件处理都已经具备。下一步最值得投入的不是继续零散加按钮，而是建立统一工具箱和图层数据模型，把高频 Web GIS 能力串成闭环。

推荐近期优先级为：

1. 图层详情 + 属性表 + 单图层导出。
2. 导入向导 + CRS/字段/范围检查。
3. Buffer + 图层统计两个轻量分析工具。
4. 图层样式 + 标签，提升制图表达。
5. Clip/Intersect/字段管理，正式进入数据处理工具箱。

这样推进后，系统会从“在线 GIS 查看器 + Mapshaper 页面”升级为“轻量、便捷、可扩展的 GIS 在线工具合集”。
